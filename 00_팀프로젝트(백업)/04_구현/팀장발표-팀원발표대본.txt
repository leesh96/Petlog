9조 펫로그의 팀장 이수호입니다. 중간발표를 진행하도록 하겠습니다.

발표순서 입니다.

저희 프로젝트는 크게 주요한 4가지의 기능을 제공하는데,
반려동물 관리 측면에서 일기작성기능과 반려동물 정보를 등록할수 있는 기능을 제공하고
반려동물 커뮤니티 측면에서 펫스타그램과 산책로, 물품나눔 게시판 기능을 제공합니다
이를 통해 반려동물인들이 흥미를 가지고 사용할 수 있는 반려동물 어플리케이션의 개발이 목표입니다.

저희 프로젝트의 구현환경입니다.
안드로이드 스튜디오 버전 3.6.3으로 어플리케이션 개발을 진행하고
개발에 이용하는 언어는 자바입니다.
안드로이드 API 는 버전 30을 사용하고 개발에 사용되는 가상에뮬레이터는 갤럭시 S20+ 기준입니다.

서버와 DB환경을 보면
웹서버를 구축하여 서버상에 Apache와 MySQL, PHP를 설치하여서 사용하고 있습니다.

설계변경 사항입니다.

기능적 관점에서의 변경사항으로 
기능2.3.2 펫스타그램 게시물 작성과 기능 1.2.1 일기작성에 있어서 미디어를 첨부할 때 사진이나 동영상 첨부를 하도록 설계하였으나
동영상을 서버에 업로드 하고 서버에 업로드 되어있는 동영상을 다시 어플리케이션으로 가져오는 작업의 개발이 난이도가 높고 시간이 오래소요되기 때문에 시간관계상 삭제 하고 사진만 첨부 하도록 변경하였습니다.

기능 6.1 구글계정 로그인과 기능 6.4.1 구글계정 회원가입 기능의 경우 개발 진행도중
이메일로 회원가입한 사람, 구글계정으로 회원가입한 사람의 정보를 개별적으로 관리해야 한다는 점과
단순히 로그인, 회원가입만 구현하는 것이 아닌 앱기능 전반적으로 이용해야하는 닉네임과 이메일주소는 개인정보의 보호문제가 겹치고
또한 구글 회원탈퇴 진행시 어플리케이션의 로그인 역시 불가능 해지기 때문에 해당 구글 계정으로 작성한 일기와 게시물들의 정보가
관리 불능 상태가 된다는 점, 별도의 파이어베이스를 사용해야 하는 점들 때문에 삭제하게 되었습니다.

정보적 관점에서의 변경사항으로
사진을 데이터베이스에 저장할 때 설계당시 생각했던 방식은  MySQL 테이블에 Blob 형태의 데이터로 직접 집어넣는 방법을 생각했습니다.
하지만 Blob 형식 이미지를 주고받을 때 6단계의 과정을 거쳐야 하고 인코딩과 디코딩 과정이 들어가기 때문에
데이터의 효율적인 관리와 실행속도의 증가를 위해서 사진파일을 서버로 직접 업로드 하고
DB테이블에는 사진파일이 있는 경로를 업로드하는 방식으로 변경하기로 하였습니다.

서버 구축상황입니다.
보이는 화면1은 구축되어진 서버에 Putty를 사용해 접속한 화면입니다.

화면3은 개발 진행 하면서 앱과 DB의 연동을 위해 작성한 php파일들이 서버에 정상 업로드 되어있는 화면입니다.

자동로그인과 비밀번호 찾기의 구현상황입니다.
사용자가 앱을 최초 실행하면 로그인 화면으로 실행됩니다.
사용자가 로그인을 하게 되면 로그인한 회원의 정보가 DB에서 넘어와 어플리케이션에 "key" = "데이터" 형태로 저장되어지고
로그아웃을 하게 되면 저장됭 회원정보를 모두 삭제합니다. 이는 앱에서만 삭제하는 것으로 DB에선 삭제되어지지 않습니다.
앱을 실행하면 저장된 회원정보가 있는지 확인하고 있으면 메인화면, 없으면 로그인화면으로 실행되어 지게 만들었습니다.

비밀번호 찾기는 이메일인증을 통해 진행됩니다.
가입 시에 이용한 이메일 주소를 입력하게 되면
Java X Mail 라이브러리를 이용하여 보낸사람은 개발자의 메일주소로 인증번호 메일이 전송됩니다.
인증번호는 알파벳 a-z 숫자0-9를 랜덤으로 추출한 8자리 문자열입니다.
보내어진 인증번호는 어플리케이션이 기억하게 되고 인증번호를 입력하고 인증번호가 맞으면 비밀번호를 변경할 수 있는 페이지로 넘어가게 됩니다.
여기서 비밀번호를 재설정 하면 user_info 테이블의 사용자 이메일주소에 해당하는 비밀번호가 변경되어 집니다.

일기 기능 구현상황 입니다.
일기를 작성할 때 일기를 작성하려는 날짜를 선택하고 반려동물의 감정이 어때보였는지 선택합니다.
반려동물의 감정은 이미지뷰를 클릭하면 선택다이얼로그가 출력되고 여기서 감정을 선택하면 알맞는 표정이 뜨게 됩니다.
제목과 내용은 텍스트 입력입니다.
작성완료 후 우상단에 체크 버튼을 클릭하면 DB에 작성한 정보가 업로드 되어짐을 알 수 있습니다.

일기 목록에서는 감정스티커, 제목, 작성날짜만 보여집니다.
목록은 리사이클러 뷰로 구현하였습니다.
test1234로 로그인 되어있기 때문에 사용자가 작성한 두개의 일기만 보여집니다.
일기 목록에서 날짜를 선택하면 해당 날짜에 작성된 일기만 보여집니다.
목록에서 클릭하면 화면4 일기 보기 화면으로 넘어가고 작성한 모든 내용을 볼 수 있습니다.

마지막으로 MyPet기능 입니다.
메인화면에서 우상단에 메뉴 버튼을 클릭하면 추가하기 수정하기 삭제하기 팝업메뉴가 뜨도록 구현하였습니다.
여기서 추가하기를 클릭하면 화면1 정보 추가 화면이 보여집니다.
여기서 위에 갤러리 이미지 뷰를 클릭하면 화면2처럼 갤러리에서 사진을 선택할 수 있습니다.
정보를 입력하고 추가하기를 클릭하면 화면3처럼 정상적으로 정보가 업로드 되어짐을 알 수 있습니다.

일기와 마찬가지로 사용자의 반려동물 정보만 보여지고 목록에서 클릭해서 상세 정보를 볼 수 있습니다.
뷰페이저로 구현하게 되면 화면5 정보보기 화면이 스와이프 되면서 여러 반려동물정보를 볼 수 있게 할 예정입니다.
반려동물 사진의 경우 DB에 BLOB 형태로 업로드 되는 것 까지는 구현하였으나 앞서 설계변경사항에 말씀드린대로 변경해서
